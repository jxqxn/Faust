# 从《密教模拟器》到《苏丹的游戏》：设计创新与内容生产管线

> 文档状态：设计参考基线
> 更新日期：2026-07-15
> 用途：辅助 Faust 从复刻研究转向原创游戏立项、核心循环评审、垂直切片规划与内容管线建设。

## 1. 文档定位

本项目的长期目标不是停留在复刻《苏丹的游戏》，而是像《苏丹的游戏》改造《密教模拟器》那样：先理解一套成熟设计为何有效，再识别它的体验边界，最终建立属于我们自己的核心命题、玩法对象和内容生产方法。

本文回答四个问题：

1. Faust 当前已经积累了哪些可复用的游戏能力？
2. 《苏丹的游戏》相较《密教模拟器》改变了什么？
3. 这些改变最可能经过怎样的策划与生产流程？
4. 我们设计原创游戏时，应该怎样复用这套方法而不是照搬表层机制？

本文不是《苏丹的游戏》开发团队的历史记录。涉及原作运行方式的内容，按逆向语料库验证；涉及创作动机、团队流程和设计目的的内容，均属于从产品结果反推的策划假说。

### 1.1 证据标签

- **已验证事实**：有反编译 `.c`、`dump.cs`、原始配置或运行观察支撑。
- **配置观察**：能够确认数据形态和内容规模，但不能单独证明运行时行为。
- **策划推断**：从最终体验和系统结构反推设计意图或生产过程。
- **待验证问题**：未来立项或实现前需要通过原型、玩家测试或进一步研究回答。

## 2. Faust 当前成果：从策划角度看我们拥有什么

### 2.1 当前产品阶段

Faust 已经从“桌面卡牌界面原型”进入“首周垂直切片成立的游戏框架”阶段。

它具备一套能持续扩展的叙事卡牌游戏底座，但尚不具备完整原作的内容覆盖。当前最准确的表述是：**规则骨架与首批内容链已经可玩，后续重点是按玩家可达性扩展内容、DSL 与表现层。**

### 2.2 已经成立的核心体验

玩家在轮次与天数压力下获得苏丹卡，并在期限内通过人物、物品、仪式、事件和资源安排解决强制目标。解决问题的过程会改变人物状态、产生新内容，并影响后续选择。

当前已具备：

- 苏丹卡牌库、抽取、重抽、期限、完成、失败和下一轮启动；
- 日历与轮次推进；
- 运行时人物/物品卡实例；
- 运行时标签、数量、丢失状态、手牌/槽位/苏丹区归属；
- 仪式实例、卡槽、开放、开始、等待、结算和清理；
- 条件、结果、动作、骰子、计数器、标签、分支和掉落；
- 事件、提示、选择和延迟操作；
- v5 存读档、继续游戏和独立手动存档；
- 主桌面、仪式浮层、事件浮层、卡牌详情和游戏结束界面。

### 2.3 已验收的内容切片

当前正常游戏接受的首周内容范围是：

- 治理家业；
- 俺寻思；
- 书店营业与淘书生成链；
- “上朝 → 权力的游戏 → 标签移除”实例链。

这些内容不是任意挑选的四段剧情，而是在验证四类生产能力：

| 内容 | 主要验证能力 |
|---|---|
| 治理家业 | 常驻仪式、自动开始、自动结算、人物投入和收益 |
| 俺寻思 | 桌面交互、主动触发和事件/掉落反馈 |
| 书店/淘书 | 掉落池、随机变体、运行时仪式生成和存档恢复 |
| 上朝/权力的游戏 | 苏丹卡实例上下文、人物标签和连续事件状态 |

### 2.4 工程治理能力

Faust 已建立两条对原创开发同样重要的边界：

1. 未支持的 DSL 必须继续进入机器可读审计，不能因为能够解析键名就宣称已经支持。
2. 配置定义不等于运行时内容；一个仪式存在于 JSON 中，不代表它应该在正常开局出现。

可达性图用于挑选下一批内容，但只能说明“静态上可能抵达”，不能代替真实运行验证。

截至 2026-07-14 的最近一次完整测试记录：13 个测试脚本、229 个测试、2952 个断言全部通过。该数字是阶段快照，后续应以最新 `gut-test.log` 为准。

### 2.5 当前尚未完成的部分

- 全量事件链和仪式内容；
- 全量 Condition/Result/Action DSL；
- 完整新手引导和长局节奏；
- 原作全部演出、地图反馈和表现层；
- 原创游戏的核心幻想、压力源、角色体系和长期结构。

因此，当前资产适合承担“原创规则实验场”，不应被误认为已经接近完整内容产品。

## 3. 《苏丹的游戏》相较《密教模拟器》的核心改造

### 3.1 总结

《密教模拟器》是一张开放式炼金台：玩家自行寻找目标，并通过不透明的卡牌组合发现世界。

《苏丹的游戏》将其改造成一张被权力驱动的调度台：系统持续提出不可拒绝的问题，玩家决定如何解决、使用谁、牺牲什么，并承担人物与政治后果。

它保留了卡牌组合叙事，却重写了三个根问题：

- 是谁向玩家提问；
- 玩家拿什么作答；
- 答案如何留在游戏世界里。

### 3.2 改造矩阵

| 维度 | 《密教模拟器》倾向 | 《苏丹的游戏》改造 | 主要解决的痛点 |
|---|---|---|---|
| 核心驱动 | 玩家自主寻找目标 | 每轮获得必须处理的苏丹卡 | 减少“现在该做什么”的空转 |
| 时间压力 | 多个实时计时器并行 | 主动推进日期，期限集中在明确任务上 | 降低计时器焦虑，允许完整规划 |
| 卡牌语义 | 概念、知识、工艺、资源 | 人物、关系、身份、物品和经历 | 让系统选择天然携带情感含义 |
| 失败结构 | 常因未知机制或计时失控失败 | 强制任务、期限与道德代价构成冲突 | 让失败更像选择的后果，而非没读懂规则 |
| 组合发现 | 高度隐秘的配方试验 | 仪式槽位、条件提示、等级与地图入口 | 保留探索，同时提高局部可读性 |
| 随机调度 | 长期不确定性较强 | 苏丹卡、骰子、重抽、金骰和情报等短期工具 | 给坏运气提供可操作的回应 |
| 叙事形态 | 玩家拼合世界观碎片 | 角色关系、政治处境和连续事件 | 更容易形成角色记忆与剧情节奏 |
| 长期状态 | 多由资源和进程表示 | 状态沉淀在具体人物和关系上 | 让过去的选择持续改变未来选项 |

### 3.3 最重要的创新：外部强制命题

苏丹卡不仅是倒计时或任务卡，而是一台持续生成戏剧问题的机器。

它把玩家的思考从：

> “我接下来可以研究什么？”

转换为：

> “这个问题我必须解决；我愿意用谁、欠下什么、冒什么风险？”

由此产生稳定的回合张力：

```text
外部不可拒绝的压力
→ 有限时间内寻找解法
→ 调度人物、关系与资源
→ 支付代价并留下状态
→ 新状态改变下一次压力的解法
```

**已验证事实：**原作中苏丹卡生成和重抽是独立的核心控制流程，而不是普通事件的文本包装。

- `[SRC: decompiled/GameController.c @ TryGenSudanCard (RVA 0x559730, dump.cs:320076)]`
- `[SRC: decompiled/GameController.c @ RedrawSudanCard (RVA 0x5558b0, dump.cs:320085)]`

### 3.4 第二个创新：把系统资源人格化

人物既是能力集合，也是关系对象和剧情记忆载体。同一人物参与过什么、获得了什么标签、是否被损失、目前位于哪个仪式，都可能改变后续内容。

这使“投入一张卡”不再只是支付数值，而可能意味着：

- 暴露一个盟友；
- 消耗一段关系；
- 把某人推向新的身份；
- 给未来事件留下证据；
- 永久失去一条解决路线。

对原创设计而言，这比复制苏丹卡的四种类型更值得学习。

### 3.5 第三个创新：把组合复杂度局部化

《密教模拟器》的复杂度分散在整张桌面和多个动词中。《苏丹的游戏》用仪式把一次决策收束为局部问题：

- 这是什么行动？
- 哪些槽位可用？
- 每个槽位接受什么？
- 何时开始和结算？
- 哪组条件命中哪个结果？

玩家面对的是一个个可理解的局部谜题，策划面对的则是一个可重复配置的决策容器。

## 4. 反推的创新生产管线

下面不是原团队史实，而是根据最终产品结构反推的高概率流程。

### 阶段 A：拆解母体

#### 目标

识别《密教模拟器》中真正需要继承的体验原子，而不是复制其全部形式。

#### 可能的交付物

一张四象限矩阵：

| 必须继承 | 必须修复 |
|---|---|
| 桌面卡牌、组合、隐藏条件、状态积累、涌现叙事 | 目标模糊、实时焦虑、试错成本、组合语义晦涩 |

| 可以替换 | 可以新增 |
|---|---|
| 抽象动词、神秘学概念、完全自主的目标寻找 | 强制任务、人物关系、政治处境、明确期限、短期随机调度 |

#### 阶段闸门

团队必须能够用一句话说明：保留了什么、替换了什么、为什么替换。

建议使用的句式：

> 我们保留 ______ 带来的体验，但把 ______ 改为 ______，从而让玩家更强烈地感受到 ______。

### 阶段 B：提出单一创新命题

#### 目标

把竞品痛点压缩成一个能够驱动全局设计的核心命题。

《苏丹的游戏》可能的命题是：

> 玩家仍通过卡牌组合发现故事，但每轮都必须处理一个来自统治者的命令；组合的主要对象是具体人物，结果会持续留在人物身上。

这个命题同时约束：

- 任务结构；
- 时间结构；
- 卡牌对象；
- 失败方式；
- 叙事状态；
- UI 信息优先级。

#### 阶段闸门

如果一个新系统无法强化这句话，就不应在第一版进入项目。

### 阶段 C：制作最小压力原型

#### 原型内容

- 一张当前命令；
- 若干人物；
- 一个期限；
- 至少两种解决方式；
- 不行动的失败结果；
- 一项会带入下一轮的后果。

最早可以使用纸面卡牌、电子表格或纯文本界面，不需要地图、长剧情和完整美术。

#### 核心测试问题

1. 玩家能否随时说出当前目标？
2. 玩家能否看见期限和合法解法？
3. 玩家是否必须在两个有吸引力但有代价的方案间权衡？
4. 玩家是否会记住自己牺牲了谁或改变了谁？
5. 上一轮的决定是否改变了下一轮的解题空间？

#### 阶段闸门

没有长文本和演出的情况下，玩家仍然能够复述一次有冲突的选择。

### 阶段 D：重建时间架构

#### 目标

将连续实时压力改造成玩家可计划、策划可预测的离散结算。

可能的规则表：

```text
玩家安排卡牌
→ 启动或等待行动
→ 主动推进一天
→ 触发轮末事件
→ 更新仪式生命周期
→ 执行延迟操作
→ 扣减强制任务期限
→ 检查失败
→ 在条件满足时进入新轮
```

#### 生产意义

这张顺序表会同时成为：

- 策划规则基线；
- 程序调用顺序；
- 叙事演出顺序；
- 存档边界；
- QA 回归用例来源。

### 阶段 E：建立可持续变化的对象模型

#### 目标

确定什么东西会承载长期记忆。

《苏丹的游戏》选择了人物实例。人物至少同时承担：

- 基础能力；
- 社会身份；
- 关系位置；
- 剧情标签；
- 当前占用状态；
- 可失去或可转化状态。

#### 策划交付物

一份“卡牌本体与运行时状态表”，明确区分：

| 定义层 | 实例层 |
|---|---|
| 名称、头像、基础属性、初始标签 | 当前标签、数量、位置、所属行动、受伤/死亡/转化状态 |

#### 阶段闸门

同一种人物定义产生两个实例时，两者可以拥有不同经历，并被后续内容分别识别。

### 阶段 F：设计通用决策容器

#### 目标

让不同题材的行动共享一套结构，降低每条剧情的程序成本。

原作仪式结构包含：

- 开放条件；
- 自动开始与自动结算标记；
- 卡牌槽位及槽位条件；
- 等待与生命周期；
- 优先、普通和额外结算；
- 结果与后续动作。

**符号事实：**这些字段存在于 `RiteNode` 数据结构中。

- `[SRC: dump.cs:393154, RiteNode]`
- `[SRC: dump.cs:393176-393208, auto_begin/auto_result/settlement/cards_slot fields]`

#### 生产原则

仪式不是一个具体玩法，而是一种“局部决策页面”。治理、上朝、淘书、谋杀、宴会等题材都可以复用，只替换槽位、条件、文本和结算。

#### 阶段闸门

策划能够仅使用已有通用规则，制作三种题材明显不同的行动。

### 阶段 G：形成三层内容语言

#### 仪式：玩家主动决策层

负责投入什么、满足什么、等待多久、得到什么结果。

#### 事件：叙事演出层

负责解释处境、呈现对话、提供选择、修改状态、开启后续内容。

#### 掉落：内容生成层

负责获得人物/物品、生成仪式、抽取剧情变体和控制重复。

三层通过标签、计数器、事件 ID、卡牌 ID 与生成关系连接：

```text
强制任务提出问题
→ 事件说明处境
→ 生成一个或多个行动
→ 玩家投入人物与物品
→ 结算修改人物、关系和资源
→ 掉落生成新的机会
→ 后续事件读取已经留下的状态
```

#### 配置观察

逆向语料库当前包含约：

- 1495 个仪式 JSON；
- 1863 个事件 JSON；
- 193 个掉落 JSON。

文件数量不能直接等同于有效玩法数量，但足以说明原作采用了“通用系统 + 大量数据配置”的内容生产方式。

### 阶段 H：用垂直切片验收管线

垂直切片不以“第一章写了多长”为标准，而要有意覆盖生产风险：

- 常驻行动；
- 主动触发行动；
- 随机生成内容；
- 连续事件与人物状态；
- 强制任务、期限与失败；
- 中途存档和恢复。

#### 阶段闸门

一条链必须满足：

> 玩家能从正常开局抵达，理解问题，做出选择，看到结果，保存并恢复状态，而且该结果会改变后续游戏。

### 阶段 I：从重复内容中提炼 DSL

#### 正确方向

先写具体内容，观察多个内容链反复需要什么规则，再把共同需求提炼为通用语法。

```text
具体内容需求
→ 两到三个重复用例
→ 定义通用语义和作用域
→ 实现执行路径
→ 加入审计白名单
→ 为边界添加测试
```

#### 应避免的方向

- 先造一套理论上万能的 DSL；
- 因为配置里出现了一个键就立即支持它；
- 用字符串前缀把未知变体整体标为支持；
- 按“未支持键数量”机械安排开发优先级；
- 把所有短跳数配置直接开放到正常游戏。

### 阶段 J：进入内容包生产

成熟后，一条内容链可以采用以下流水线：

1. **命题提案**：玩家面临什么问题，涉及谁，为什么现在必须处理。
2. **选择结构**：至少两种解决路线，以及不行动的结果。
3. **状态设计**：列出读取和写入的标签、资源、计数器与人物状态。
4. **依赖图**：标明入口事件、生成仪式、掉落和后续出口。
5. **配置实现**：只使用已支持的条件、结果和动作语法。
6. **静态审计**：检查未知 DSL、悬空 ID、不可达内容和错误作用域。
7. **确定性测试**：固定随机种子，验证成功、失败、特殊组合和存读档。
8. **体验验收**：检查目标理解、代价预判、结果记忆和后续影响。

一个内容包的验收单位不应是“写完 20 个事件”，而应是“一条完整可达、可选、可结算、可延续的玩家经历”。

## 5. 这套创新为何能够量产

整套设计经历了两次“编译”。

### 5.1 第一次：把体验痛点编译成机制

| 痛点 | 机制回应 |
|---|---|
| 不知道做什么 | 强制任务/苏丹卡 |
| 多计时器焦虑 | 主动推进日期 |
| 卡牌过于抽象 | 具体人物与关系 |
| 失败原因不明 | 明确任务、期限和不行动后果 |
| 随机无法干预 | 重抽、金骰、情报等调度工具 |
| 故事状态难感知 | 人物标签、身份和关系变化 |

### 5.2 第二次：把机制编译成生产工具

| 设计需求 | 内容工具 |
|---|---|
| 人物参加行动 | 仪式槽位 |
| 判断人物经历 | 标签与作用域条件 |
| 表达多种结果 | 分层结算和分支 |
| 延续剧情 | 事件与延迟操作 |
| 生成新机会 | 掉落系统 |
| 批量检查 | DSL 审计、可达性分析和自动化测试 |

真正成熟的创新不仅要在玩家第一次看到时显得新鲜，还必须能让团队稳定地生产、验证和维护大量内容。

## 6. 对我们原创游戏的直接启发

### 6.1 不要复制表层公式

不应把以下元素直接当成原创方案：

- 换皮的苏丹卡；
- 四种任务类型与四档稀有度；
- 同样的宫廷人物；
- 同样的仪式与骰子；
- 同样的日历节奏。

真正应该继承的是深层结构：

```text
外部不可拒绝的压力
→ 有限的应对空间
→ 对有情感意义对象的调度
→ 可记忆、可持续的后果
→ 被后果改变的下一次压力
```

### 6.2 原创设计首先要回答的三个问题

#### 谁在向玩家施压？

它可能是统治者、战争、债务、组织、生态灾难、舆论、神明、家族义务或玩家自己的身体。

必须说明：为什么玩家不能简单拒绝？

#### 玩家拿什么作答？

是人物、城市、舰队、记忆、时间、信任、身份、领土，还是身体部位？

这些对象必须既有系统价值，也有情感价值。

#### 答案如何留下？

结果应尽量改变未来的可选项，而不仅是增加或减少通用资源。

玩家应当能说出：

> “因为我上次做了那件事，所以这个人、这个地方或这个组织现在变成了这样。”

### 6.3 推荐的原创开发顺序

```text
竞品痛点表
→ 一句话体验命题
→ 不可拒绝的压力原型
→ 时间结构原型
→ 可持续变化的对象模型
→ 通用决策容器
→ 3～5 条垂直内容链
→ 从重复需求中提炼 DSL
→ 内容包生产
→ 可达性、存档和回归验收
```

最重要的纪律是：**先用具体内容证明一个新规则值得存在，再把重复出现的规则抽象成系统。**

## 8. 后续设计评审建议

未来评审原创方案时，建议固定使用以下顺序：

1. 先评核心幻想与压力来源，不先评功能数量。
2. 再评玩家调度的对象是否具有情感意义。
3. 再评一次选择能否改变未来，而非只改变资源数字。
4. 用三条具体内容链验证系统需求。
5. 只有重复出现的需求才进入通用规则或 DSL。
6. 以完整玩家经历安排内容包，而不是按配置数量安排。
7. 每个里程碑同时通过体验验收、可达性检查、存档验证和回归测试。

本文后续应随着原创方向的确定继续拆分为：

- 原创游戏愿景与设计支柱；
- 核心循环与时间结构；
- 角色/对象本体；
- 压力卡或命题生成系统；
- 垂直切片内容图；
- 内容 DSL 与制作规范；
- 平衡指标与玩家测试计划。
